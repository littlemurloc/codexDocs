# UI/UX 设计原则

本文档记录《三国斗阵》当前 UI/UX 的全局设计原则。具体界面图、资源状态和版本记录放入 [asset-manifest.md](asset-manifest.md)，单张图的设计意图放入对应说明文档。

## 总体方向

- UI 服务于“战前判断、波次后抉择、战斗观赏、局外成长”，不做过重装饰。
- 视觉方向参考《三国志14》的现代历史策略界面：清爽、克制、信息层级明确。
- 三国题材主要体现在地图纹理、墨色线条、朱红/金铜点缀、武将头像、阵营与兵种图标上。
- 面板应轻量、扁平、半透明或浅色石瓷质感，边框细，少用厚重拟物。
- 中文 UI 文案必须短、清楚、可读，避免长说明塞进界面。

## 明确规避

- 不做页游大厅风：不要黑金充值感、礼包入口堆叠、活动弹窗、夸张发光按钮。
- 不做过暗压抑风：除战斗场景自身需要外，界面底色不要大面积黑蓝或暗金。
- 不回到厚重历史道具风：卷轴、竹简、羊皮纸、雕花框只能作为轻微点缀，不作为主体界面语言。
- 不用营销落地页结构：首屏必须是可操作游戏界面，不是宣传海报。
- 不用重复主操作：同一界面中相同功能不要出现两个同权重按钮。
- 不让 UI 遮挡核心玩法观赏：战斗界面应优先保留战场空间。

## 关键玩法约束

- 出战结构为默认 2 人，最多 4 人，设计上按 `2 + 2` 理解。
- 4 人结构应明确支持 `主将 / 主将 / 副将 / 副将` 或等价槽位表达。
- 未解锁或暂未上阵的槽位可弱化预留，但不要在战斗 HUD 中用醒目的锁、文字或空状态抢占注意。
- 关卡不是强制覆盖式推进。已解锁关卡可以回打。
- 关卡切换优先使用同层级地图节点，不使用深层级列表或下拉作为主路径。
- 局内波次结束后的战备是 Roguelike 运行内能力选择，需要清楚表达“暂停、预览下一波、三选一、继续战斗”。

## 局内信息优先级

1. 当前波次、战斗状态和胜负风险。
2. 我方武将生命、关键状态、技能冷却和基础属性。
3. 敌方关键单位、Boss 阶段和危险技能前摇。
4. 已选择的本局战备能力。
5. 羁绊与装备带来的流派强化。

## 战斗 HUD 原则

- 战场画面应采用俯视或 3/4 俯视的 RTS / 自走棋观战感，避免定点回合制站桩感。
- 角色单位信息放在底部居中区域，默认按最多 4 槽设计。
- 单个单位信息优先采用窄竖卡，横向节省空间，适配立绘背景。
- 每名上阵角色需要表达：名字、血量/能量、4 个技能、2 个装备、攻、防、攻速、移速。
- 默认 2 人时也要为最多 4 人预留布局空间，但未启用槽位应尽量安静。

## 3x3 站位与选敌表现

- 布阵和备战拖拽英雄时，当前格位所在行应高亮，并用目标线显示该格优先攻击的敌方行。
- 主目标线使用较清晰的实线；次级目标顺序使用更淡的虚线，避免和主目标混淆。
- 上路、中路、下路应有短标签解释目标逻辑，例如“优先上路，空则中路、下路”“优先中路，空则支援敌多一路”。
- 中路无优先目标时，UI 应表达“敌多优先，数量相同随机”，避免玩家误以为中路完全随机。
- 拖起拥有站位被动的英雄时，可触发被动的格位应高亮；落位后用短提示、角标或格位纹样说明已激活被动。
- 战斗中不常驻显示所有目标线。只在开战前短暂展示对位线、单位切换目标时闪现目标线、或玩家悬停/长按单位时显示该单位目标逻辑。
- 当前攻击目标可用目标脚下圈、短脉冲线或轻量锁定标识表达；无目标转向时可短暂显示“支援中路”“支援下路”等提示。

## 备战状态原则

- 备战状态必须清楚表达“战斗暂停，可以做决策”。
- 下一波敌人预览应先展示敌人类型、关键词压力和特殊机制，再展示细节数值。
- 战备抉择卡片应让玩家一眼看出影响对象、关键词、持续时间和适配流派。
- 右侧摘要区不是阵容编辑面板，而是本局已选战备、当前流派倾向、当前队伍摘要。
- 当前队伍摘要只做轻量头像/标签提醒，不展开技能、装备和详细属性。

## 图片资源规则

- 只有已获得项目负责人或主策划审核认可的图片才存档保留到 `assets/ui/preview/`。
- 未通过审核的迭代图只作为临时预览，不登记为正式资源；若已经误存，应在最终版确认后清理。
- 正式导出资源放入 `assets/ui/export/`。
- 源文件放入 `assets/ui/source/`。
- 每个已确认概念图都应在 [asset-manifest.md](asset-manifest.md) 中登记，并配套一份说明文档。

## 与主策划协作

- UI/UX 可以提出信息结构优化建议。
- 如果 UI 方案需要改变玩法流程、奖励时机、战备抉择规则或商业化入口，应先在对话中请求确认。
- 主策划只在被指定时评审具体 UI 图片，默认通过索引和文字说明理解 UI 进度。

## 效果图产出流程

- 在产出下一张 UI 效果图前，UI/UX 必须先说明：准备做的界面名称、该界面在游戏流程中的展示时机、它承载的核心信息与不承载的内容。
- 只有在项目负责人确认该界面方向后，才进入效果图生成。
- 未经确认直接生成的图片只视为临时草稿，不存入 `assets/ui/preview/`，不登记到 `asset-manifest.md`。
